20210330_java_加密加签

sm4 sm2 AES RSA
java
发布日期:   2021-03-30

引言：这篇文章主要用来记录我的工作心得，当一个APP运营到一定规模的时候，会考虑引入其他的第三方APP来达到客户引流的目的，进一步增加APP的日活和月活（日活跃人数和月活跃人数）

​ 在我负责的对接第三方过程，与支付宝的过程类似，都用到了对称加密和非对称加密数据的算法，关于这两个算法我先简要介绍一下，对称加密算法就是同一个秘钥既可以用于加密，也可以用于解密，可以实现双向的数据转换，这也是一种可逆算法。

​ 国际上最常见的对称加密算法是AES(Advanced Encrypt Standard 高级加密标准)，最常见的非对称加密算法是RSA(以当初发明这种算法的三名科学家的名字首字母命名)，RSA采用的秘钥越长越安全，目前公认的是2048位及以上的秘钥是安全的。这两种算法的具体实现方式可以去网上搜，各种编程语言的实现版本都有。

​ 这里要说一下国内通用的加密算法，也就是SM2(非对称加密)，SM4(对称加密)，SM3(…国产哈希算法)，至于具体的用法，下面以我自身的经验出发进行讲解

​ 首先针对与第三方进行交换的数据，为了避免数据在网络传输过程中被劫持从而导致信息泄露，需要对数据进行加密，这是生产上所有APP都必须重视的一个问题，不仅如此，还要防止数据在传输过程中被篡改，要对传输数据进行加签验证数据的准确性。

​ 因为要使用非对称加密算法，所以假设我的手上有一对公私钥c_public和c_private，等待交互的第三方手上也有一对公私钥d_public和d_private。现在有一个源数据串A需要发送给第三方，需要先对源数据进行加签，使用SM3算法生成签值B并且拼接在源数据后面形成新的数据串AB(SM3算法并不需要秘钥)，再针对这串数据使用第三方公钥d_public和SM4加密生成加密后的字符串cipher以及对应的秘钥key和偏移量iv，最后使用第三方公钥d_public和SM2针对key和iv分别进行加密得到new_key和new_iv，将这三个值cipher、new_key、new_iv采用https协议通过互联网送到第三方手中，而第三方解密的过程就是一个对应的反过程。先使用第三方的私钥d_private针对new_key和new_iv进行SM2解密得到key和iv，然后根据第三方公钥d_public和SM4算法以及cipher、key、iv解密得到原本的字符串AB，最后是验签的过程，因为SM3算法是一个不可逆算法，所以使用SM3对A串再进行加密得到一个new_B数据串，比较B和new_B，如果两个值相等就表示验签成功，数据在传输过程中没有发生更改。第三方可以信任收到的数据并进行后续的业务逻辑处理。
